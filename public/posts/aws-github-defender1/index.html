A Day in the Life of a Cloud Security Engineer: Mitigating AWS & GitHub Risks with Microsoft Defender for Cloud#

Cloud environments are dynamic—and so are their risks. As a security engineer, you’re responsible not just for identifying misconfigurations and vulnerabilities, but also for prioritizing, communicating, and fixing them before they become incidents.

This post walks you through how I use Microsoft Defender for Cloud, particularly Cloud Security Posture Management (CSPM) and Cloud Workload Protection (CWP), to secure AWS and GitHub assets. We’ll explore how Defender helps surface issues like open SSH ports, missing VPC flow logs, public S3 buckets, exposed secrets in GitHub—and how to fix them.

Onboarding AWS & GitHub: Quick Summary#

Before jumping into detections, here’s a quick view of the integration:

  • AWS: I onboarded AWS via the native connector using a cross-account IAM role with the SecurityAudit and ReadOnlyAccess policies.

  • GitHub: I connected the org repo via OAuth with scoped access to scan for secrets and configuration drift.

Once Defender starts scanning, vulnerabilities are automatically surfaced in the Recommendations blade.

Risk Detection with Defender CSPM#

After 24 hours, Defender CSPM populated over 290 posture recommendations across AWS, most in the Low to Medium risk range.

Here are some real-world detections from my test AWS tenant:

Open SSH Ports (0.0.0.0/0 on Port 22)#

  • Detection: Security groups sg-0d45a032c0d2a23ef allow unrestricted SSH access.

  • Risk: High—exposes EC2 instances to brute-force or credential stuffing attacks.

  • Remediation:

    • Restrict ingress rules to a known IP or bastion host.
    • Use Systems Manager (SSM) Session Manager instead of SSH.

Public or Unrestricted S3 Buckets#

  • Detection: insecure-bucket-akshay lacks block public access.
  • Risk: Sensitive data leakage.
  • Remediation:
    • Enable block public access on the bucket.
    • Review bucket policies and ACLs for anonymous access.

aws s3api put-public-access-block \
--bucket insecure-bucket-akshay \
--public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

Always set this on buckets storing sensitive data, logs, backups, or anything internal.

Unencrypted S3 Buckets#

  • Detection: unencrypted-bucket-akshay was flagged
  • Risk: Data exposure if intercepted or compromised.
  • Remediation:
    • Apply bucket-level or object-level encryption.
    • Enforce default encryption using policies.

Enforce default AES256 encryption on a bucket -

aws s3api put-bucket-encryption \
--bucket unencrypted-bucket-akshay \
--server-side-encryption-configuration '{
"Rules": [
{
"ApplyServerSideEncryptionByDefault": {
"SSEAlgorithm": "AES256"
}
}
]
}'

VPC Flow Logs Disabled#

  • Detection: Multiple VPCs (vpc-0820a33fab7463213, etc.) lacked flow logging.
  • Risk: Inability to trace suspicious traffic or lateral movement.
  • Remediation:
    • Enable VPC flow logging via the console or CLI.
    • Store logs in a secure S3 bucket for centralized monitoring.

aws ec2 create-flow-logs \
--resource-type VPC \
--resource-ids vpc-xxxx \
--traffic-type ALL \
--log-group-name VPCFlowLogs \
--deliver-logs-permission-arn arn:aws:iam::xxx:role/FlowLogRole

Prioritizing with Defender CSPM Risk Graph#

What sets Defender apart is its risk-based prioritization. Rather than blindly fixing 294 issues, it surfaces the most exploitable ones first, based on:

  • Attack paths
  • Public exposure
  • Resource sensitivity

You fix what matters most, first.

CWP in Action: Workload Protection & Agentless Scanning#

Enabling Cloud Workload Protection allowed me to get even deeper insights into:

  • Vulnerable OS packages on EC2 instances.
  • Exposed metadata services (e.g., IMDSv1).
  • Overprivileged IAM roles with full admin access.
  • Outdated kernel packages flagged for CVEs.

All without deploying agents—thanks to agentless scanning via snapshot inspection.

GitHub Risk Detection#

Connecting GitHub provided insight into:

  • Exposed secrets (e.g., AWS access keys in commits).
  • Over-permissive repositories (e.g., “Everyone” with write access).
  • Security policy violations (e.g., missing branch protection rules).

💡 Remediation Steps:

  • Rotate exposed credentials immediately.
  • Apply GitHub Actions secrets scanning and branch protection.
  • Use GitHub Advanced Security or third-party secret scanners for deeper protection.

Final Thoughts#

Microsoft Defender for Cloud offers a unified, risk-driven view into your cloud security posture—across AWS, Azure, GitHub, and more.

From open SSH ports to secret leaks in GitHub, it’s not just about finding issues, but prioritizing and fixing the ones that matter most.

This is how modern security engineering works: data-driven, automated, and integrated.
